## 역할 정의

### 관리자

- 정부/공공기관 이메일 인증을 거쳐 회원가입하는 계정이다.
- 축제를 생성할 수 있으며, 축제를 생성한 관리자는 해당 축제의 총괄관리자가 된다.
- 총괄관리자는 해당 축제의 운영자 초대/관리와 행사 정보 수정 권한을 가진다.
- 관리자 계정 자체는 로그인 시점에 총괄관리자 또는 운영자로 고정되지 않는다.
- 특정 축제에 대해 어떤 권한을 갖는지는 축제별 권한 관계로 판단한다.

### 운영자

- 기존 대화 기준으로 제2 관리자 또는 서브관리자에 해당한다.
- 총괄관리자가 특정 축제에 초대하거나 등록한 계정이다.
- 관리자가 수행하는 주요 운영 기능을 대부분 공유하지만, 총괄관리자 전용 권한은 제한된다.
- 총괄관리자 전용으로 분리할 기능은 운영자 초대/삭제, 축제 핵심 정보 수정, 축제 삭제 같은 권한 변경 또는 행사 설정 변경 기능이다.

### 스태프

- 일반 관리자 회원가입 절차를 거치지 않는다.
- 관리자 또는 운영자가 행사 운영을 위해 별도 계정을 생성해 부여한다.
- 앱 로그인 시 제공받은 아이디, 비밀번호, 사용자명, 전화번호 기반으로 접근한다.
- 유효 기간은 행사 시작 전 준비 기간부터 행사 마감일까지로 관리한다.
- 권한은 현장 운영에 필요한 최소 범위로 제한하며, 현재 기준 핵심 권한은 축제 부스맵 조회와 부스 줄끝라인 수정이다.

## 기능 상세

### 관리자 회원가입

- 역할: 관리자
- 카테고리: 인증
- 서브 카테고리: 회원가입
- 원문 설명: 사용자는 공무원 이메일 인증을 통해 계정을 생성할 수 있다.
- 입력 정보: 이메일, 이름, 조직, 비밀번호, 비밀번호 확인
- 이메일 조건: 정부/공공기관 이메일 도메인을 기본 허용한다.
- 현재 개발 예외: 사설 계정 테스트를 위해 `naver.com` 허용 TODO가 남아 있을 수 있으며, 실제 운영 전 제거해야 한다.
- 처리 기준:
  - 이메일 형식을 검증한다.
  - 허용 도메인인지 검증한다.
  - 이메일 인증코드를 발송하고 Redis 기반으로 인증 상태를 관리한다.
  - 비밀번호와 비밀번호 확인이 일치해야 한다.
  - 비밀번호는 해시 VO로 저장한다.
  - 중복 이메일 가입을 막는다.
- 결과: 인증된 관리자 계정이 생성된다.
- 권한 주의: 가입만으로 특정 축제의 총괄관리자/운영자가 되는 것은 아니다.

### 관리자 로그인

- 역할: 관리자
- 카테고리: 인증
- 서브 카테고리: 로그인
- 원문 설명: 사용자는 생성된 계정을 통해 로그인할 수 있다.
- 입력 정보: 이메일, 비밀번호
- 처리 기준:
  - 이메일로 관리자 계정을 조회한다.
  - 비밀번호 해시를 검증한다.
  - 성공 시 JWT를 발급한다.
  - JWT에는 계정 식별자와 인증 정보를 담고, 축제별 권한은 요청 처리 시 별도 검증한다.
- 결과: 관리자 API 접근에 사용할 JWT가 반환된다.
- 권한 주의: 로그인 시점에 제1 관리자/제2 관리자가 확정되지 않는다. 축제 생성자 여부 또는 초대 관계로 축제별 권한을 판단한다.

### 관리자 회원탈퇴

- 역할: 관리자
- 카테고리: 인증
- 서브 카테고리: 회원탈퇴
- 원문 설명: 사용자는 생성된 계정을 삭제할 수 있다.
- 현재 임시 구현:
  - 실제 DB row를 삭제하지 않는다.
  - `PATCH /api/admin/me/withdrawal` 요청으로 본인 계정 상태를 `DELETED`로 변경한다.
  - `DELETED` 상태 계정은 로그인과 인증 기반 API 사용 대상에서 제외한다.
- 처리 기준:
  - 로그인한 관리자 본인만 탈퇴할 수 있다.
  - 총괄관리자로 관리 중인 축제가 있으면 탈퇴 가능 여부 정책을 별도로 결정해야 한다.
  - 운영자 또는 스태프 초대 관계가 남아 있는 경우 정리 정책이 필요하다.
- 수정 예정:
  - 총괄관리자가 탈퇴할 때 축제 권한을 이전할지, 탈퇴를 차단할지, 축제를 비활성화할지 확정한다.
  - 운영자 권한 관계와 스태프 계정 처리 범위를 확정한다.
  - 탈퇴 후 동일 이메일 재가입 허용 여부를 확정한다.
  - 탈퇴 상태 계정의 기존 JWT를 즉시 무효화할지, 만료까지 둘지 확정한다.

### 관리자 축제리스트 조회

- 역할: 관리자
- 카테고리: 홈
- 서브 카테고리: 축제리스트
- 원문 설명: 사용자는 진행예정/진행중/진행완료된 축제리스트를 확인할 수 있다.
- 조회 기준:
  - 로그인 계정이 총괄관리자 또는 운영자로 연결된 축제를 조회한다.
  - 축제 기간을 기준으로 진행예정, 진행중, 진행완료 상태를 계산한다.
  - 같은 축제명이 여러 해에 반복될 수 있으므로, 축제 연도/기간은 개별 축제 단위로 관리한다.
- 결과 예시:
  - 진행예정 축제 목록
  - 진행중 축제 목록
  - 진행완료 축제 목록
- TODO: 목록 정렬 기준을 시작일 오름차순/내림차순 중 하나로 확정한다.

### 축제등록

- 역할: 관리자
- 카테고리: 축제등록
- 서브 카테고리: 축제등록
- 원문 설명: 사용자는 축제를 생성할 수 있다. 축제를 생성한 사용자는 해당 축제의 [총괄관리자] 권한을 갖는다.
- 입력 정보 후보:
  - 축제명
  - 개최 연도 또는 행사 기간
  - 시작일, 종료일
  - 장소
  - 설명
  - 운영 시간
  - 대표 이미지 또는 안내 자료
- 처리 기준:
  - 축제는 외부 식별자로 UUID를 사용한다.
  - 같은 축제라도 연도별 행사는 별도 축제 레코드로 저장한다.
  - 생성자는 해당 축제의 총괄관리자 권한으로 연결한다.
  - 축제 시리즈 또는 묶음 조회가 필요하면 같은 이름/시리즈 식별자로 연결한다.
- TODO: 실제 축제 등록 필드는 추후 DB 설계와 API 화면 명세를 보고 확정한다.

## 구현 시 우선순위 후보

1. 관리자 인증: 회원가입, 이메일 인증, 로그인, JWT 발급
2. 축제 생성: 생성자를 총괄관리자로 연결
3. 축제별 권한: 총괄관리자/운영자 권한 관계 저장
4. 축제 목록/상세: 접근 가능한 축제 조회와 기본정보 조회
5. 운영자 관리: 총괄관리자 전용 운영자 등록/삭제
6. 스태프 관리: 관리자/운영자용 스태프 계정 생성/삭제
7. 스태프 인증: 축제 종속 계정 로그인과 제한된 JWT
8. 부스맵/줄끝라인: 스태프가 수정 가능한 현장 데이터
9. 대시보드/혼잡도: 관리자/운영자/스태프 조회 범위 분리
10. 결과리포트: 진행완료 축제와 방문인원 입력 기반 조회
