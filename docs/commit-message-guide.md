# Commit Message Guide for Coding Agents

## 1. 목적

이 문서는 Coding Agent가 다음을 일관되게 수행하도록 한다.

- 변경 내용을 의미 단위로 분석
- atomic commit 단위로 분리
- 한국어 Conventional Commit 메시지 작성
- 사용자의 명시적 요청이 있을 때만 실제 커밋 수행

---

## 2. 최우선 원칙

아래 규칙은 다른 규칙보다 우선한다.

1. 사용자가 명시적으로 요청하지 않으면 커밋하지 않는다.
2. 하나의 커밋에는 하나의 목적만 담는다.
3. 각 커밋은 단독으로 빌드·테스트 가능해야 한다.
4. 각 커밋은 독립적으로 revert·cherry-pick 가능해야 한다.
5. 커밋 메시지는 한국어로 작성한다.
6. 제목에는 내부 작업 번호나 추적용 식별자를 넣지 않는다.
7. 제목만으로 충분하면 본문을 작성하지 않는다.
8. 파일 목록, 테스트 로그, AI 생성 문구는 커밋 메시지에 넣지 않는다.

---

## 3. 커밋 수행 조건

### 커밋해야 하는 경우

사용자가 다음과 같이 실제 커밋 수행을 명시한 경우에만 커밋한다.

- `커밋해줘`
- `commit 해줘`
- `커밋하고 push 해줘`
- `변경 사항을 커밋해줘`

### 커밋하면 안 되는 경우

다음 요청은 커밋 수행 요청이 아니다.

- 코드 리뷰
- 변경 확인
- 테스트 실행
- 수정 제안
- 리팩터링 방향 검토
- 변경 내역 정리
- 커밋 메시지 추천

이 경우에는 변경 내용과 검증 결과만 보고한다.

---

## 4. 커밋 전 판단 절차

커밋 요청을 받으면 아래 순서로 처리한다.

### Step 1. 변경 사항 확인

```bash
git status
git diff
git diff --staged
```

확인 항목:

- 변경 목적이 몇 개인가
- 기능, 버그 수정, 리팩터링, 문서, 설정 변경이 섞여 있는가
- 테스트가 구현과 함께 포함되어 있는가
- 생성 파일이나 불필요한 파일이 포함되어 있는가

### Step 2. 의미 단위 분리

두 개 이상의 목적이 섞여 있으면 커밋을 분리한다.

잘못된 예:

```text
feat(search): 검색 기능과 배포 설정 및 문서 추가
```

올바른 예:

```text
refactor(search): 검색 인덱스 설정 정리
feat(search): 검색 조건 필터링 추가
ci(deploy): 검색 서비스 배포 단계 추가
docs(search): 검색 정책 문서 추가
```

### Step 3. 의존성 순서 결정

다음 순서로 커밋한다.

1. 공통 schema, helper, util
2. 도메인 또는 핵심 로직
3. API, UI, 외부 연동
4. 배포 및 설정
5. 문서

후속 커밋이 선행 커밋 없이는 빌드되지 않는 구조는 허용할 수 있지만,
각 커밋 시점의 저장소는 항상 빌드 가능해야 한다.

### Step 4. staging 분리

필요에 따라 다음 명령을 사용한다.

```bash
git add <file>
git add -p
git restore --staged <file>
```

한 파일 안에 여러 목적의 변경이 섞여 있으면 `git add -p`로 hunk 단위 분리한다.

### Step 5. 검증

각 커밋 직전에 관련 빌드와 테스트를 실행한다.

검증 실패 상태에서는 커밋하지 않는다.

---

## 5. 기본 형식

```text
<type>[optional scope]: <한국어 설명>

[optional body]

[optional footer]
```

예시:

```text
fix(auth): 만료된 토큰 갱신 실패 처리 추가
```

```text
feat(order): 주문 취소 사유 저장 기능 추가

운영자가 취소 원인을 추적할 수 있도록
선택한 취소 사유를 주문 이력에 함께 저장한다.
```

---

## 6. 허용 타입

| 타입 | 사용 기준 |
| --- | --- |
| `feat` | 새로운 기능 또는 동작 추가 |
| `fix` | 잘못된 동작이나 버그 수정 |
| `docs` | 문서만 변경 |
| `style` | 동작 변화 없는 포맷 수정 |
| `refactor` | 기능 변화 없는 구조 개선 |
| `perf` | 성능 개선 |
| `test` | 테스트 추가 또는 수정 |
| `build` | 빌드, 의존성, 패키징 변경 |
| `ci` | CI/CD 설정 변경 |
| `chore` | 기타 유지보수 |
| `revert` | 이전 커밋 되돌리기 |

### 타입 선택 우선순위

변경 결과를 기준으로 선택한다.

- 새 동작이 생김 → `feat`
- 잘못된 동작을 바로잡음 → `fix`
- 동작은 같고 구조만 변경 → `refactor`
- 실행 속도나 자원 사용 개선 → `perf`
- 문서만 변경 → `docs`
- 테스트만 변경 → `test`
- 의존성, 빌드 설정 변경 → `build`
- 배포 파이프라인 변경 → `ci`
- 위 항목에 해당하지 않는 유지보수 → `chore`

파일 종류가 아니라 변경 목적을 기준으로 판단한다.

## 7. scope 작성 규칙

scope는 변경 영역이 명확할 때만 사용한다.

좋은 예:

```text
feat(order): 주문 취소 사유 저장 기능 추가
fix(auth): 만료 토큰 재발급 실패 처리
docs(api): 인증 API 예시 보완
```

scope를 생략해도 되는 경우:

- 저장소 전체에 영향을 주는 변경
- 자연스러운 scope를 정하기 어려운 변경
- scope가 제목 이해에 도움이 되지 않는 경우

억지로 scope를 만들지 않는다.

---

## 8. 제목 작성 규칙

### 필수 규칙

- 한 줄로 작성
- 가능하면 50자 안팎
- 마침표 사용 금지
- 무엇이 바뀌었는지 직접 표현
- 한국어 사용
- 파일 목록 나열 금지
- 내부 작업 코드와 티켓 번호 사용 금지
- 모호한 단어만 사용 금지

### 좋은 예

```text
fix(payment): 결제 승인 실패 시 중복 재시도 방지
refactor(user): 권한 검증 로직 분리
chore(infra): 설정 스냅샷 검증 범위 보완
```

### 나쁜 예

```text
fix: 이것저것 수정
chore: 작업 반영
chore: src/a.ts와 src/b.ts 수정
fix(payment): TASK-17 처리
```

### 제목 생성 공식

```text
<변경 대상> + <변경 결과>
```

예시:

```text
상품명 누락 시 기본 표시값 적용
실패 작업만 재처리하도록 배치 조건 변경
사용자 권한 검증 책임 분리
```

---

## 9. 내부 식별자 처리

다음 값은 제목에 넣지 않는다.

- 작업 번호
- 운영 티켓 번호
- 품질 규칙 번호
- 내부 단계 번호
- 외부 독자가 의미를 알 수 없는 약어

나쁜 예:

```text
chore(infra): QG-17 기준 반영
```

좋은 예:

```text
chore(infra): 설정 검증 누락 항목 보완
```

식별자가 추적에 필요하면 footer 또는 본문에서만 사용한다.

```text
Refs: ONE-123
```

확실하지 않은 식별자는 만들지 않는다.

---

## 10. 본문 작성 여부

### 본문을 생략하는 경우

제목만으로 다음 내용이 명확하면 본문을 작성하지 않는다.

- 무엇이 바뀌었는가
- 변경 목적이 무엇인가
- 별도 위험이나 마이그레이션이 없는가

예시:

```text
fix(cart): 품절 상품 수량 변경 시 오류 방지
```

### 본문을 작성하는 경우

다음 중 하나라도 해당하면 본문을 작성한다.

- 변경 이유가 제목만으로 부족함
- 기존 동작과 새 동작의 차이가 중요함
- 운영 위험이 있음
- 마이그레이션이 필요함
- 호환성 영향이 있음
- breaking change가 있음
- 대안이나 제약을 짧게 남겨야 함

---

## 11. 본문 작성 규칙

본문에는 필요한 정보만 남긴다.

### 포함할 내용

1. 왜 변경했는가
2. 무엇이 달라졌는가
3. 어떤 위험이나 호환성 영향이 있는가

### 제외할 내용

- 파일별 변경 목록
- diff 전체 요약
- 테스트 로그 전체
- 내부 추론 과정
- 작업 과정 일지
- `요청에 따라`
- `작업 완료`
- 장문의 설계 설명
- 수치와 항목 개수 나열
- AI 또는 도구 이름

### 길이 기준

- 권장: 5~10줄
- 최대: 15~20줄
- 최대 약 700자
- bullet은 3~5개
- 세부 설계 근거는 별도 문서에 작성

### 줄바꿈 기준

- 제목과 본문 사이에 빈 줄 추가
- 의미 단락 사이에 빈 줄 추가
- 한 줄은 약 72자 이내
- 한국어는 약 36자 안팎 권장
- 여러 항목은 bullet로 분리
- 코드 식별자는 백틱 사용

좋은 예:

```text
fix(product): 상품명 누락 시 목록 표시 오류 수정

상품 요약 응답에 `displayName`이 없으면
목록 렌더링이 실패하는 문제를 해결한다.

- `displayName`을 우선 사용
- 없으면 `name`을 사용
- 둘 다 없으면 기본 문구 표시

API 응답 형식은 변경하지 않는다.
```

---

## 12. footer 작성 규칙

footer는 다음 경우에만 사용한다.

### Breaking Change

```text
feat(api)!: 주문 상태 응답 필드명 변경

BREAKING CHANGE: `orderStatus` 필드를 `status`로 변경한다.
```

### 이슈 연결

```text
fix(login): 비활성 계정 로그인 차단

Refs: ONE-123
```

이슈 번호가 확실하지 않으면 footer를 작성하지 않는다.

---

## 13. Atomic Commit 규칙

### 하나의 커밋에 함께 포함할 것

- 하나의 기능 구현과 해당 테스트
- 하나의 버그 수정과 회귀 테스트
- 하나의 리팩터링과 관련 테스트 수정
- 하나의 설정 변경과 해당 검증 코드

### 분리해야 할 것

- 기능 추가와 무관한 리팩터링
- 버그 수정과 문서 대규모 개편
- 코드 포맷팅과 기능 변경
- 의존성 업그레이드와 기능 구현
- 서로 독립적으로 revert 가능한 기능
- 공통 util 추가와 여러 기능 적용

### 분리 기준 질문

아래 질문 중 하나라도 `예`이면 분리를 검토한다.

- 이 변경만 독립적으로 revert할 수 있는가
- 이 변경만 다른 브랜치에 cherry-pick할 수 있는가
- 변경 목적을 하나의 문장으로 설명하기 어려운가
- 서로 다른 Conventional Commit 타입이 필요한가
- 일부 변경만 롤백할 가능성이 있는가
- 리뷰어가 서로 다른 관점으로 검토해야 하는가

---

## 14. 구현 코드와 테스트

기능 구현과 해당 테스트는 같은 커밋에 포함한다.

좋은 예:

```text
feat(order): 주문 취소 사유 저장 기능 추가
```

포함 내용:

- 주문 취소 사유 필드
- 저장 로직
- 관련 단위 테스트
- 관련 통합 테스트

테스트만 후속 커밋으로 분리하면
중간 커밋이 검증되지 않은 상태가 되므로 피한다.

단, 기존 기능에 대한 테스트만 추가하는 작업은 `test`로 별도 커밋할 수 있다.

```text
test(order): 주문 취소 예외 시나리오 보완
```

---

## 15. 공통 코드 선반영

여러 기능에서 사용하는 공통 요소는 먼저 분리한다.

예시 순서:

```text
1. refactor(common): 날짜 범위 검증 유틸리티 분리
2. feat(report): 보고서 조회 기간 검증 추가
3. feat(export): 내보내기 기간 검증 추가
```

공통 코드와 모든 사용처를 하나의 대형 커밋으로 묶지 않는다.

단, 공통 코드가 특정 기능에서만 사용되고 독립적 가치가 없다면
기능 커밋에 함께 포함할 수 있다.

---

## 16. 커밋 메시지 생성 절차

Coding Agent는 아래 절차로 메시지를 생성한다.

### 1. 변경 목적 한 문장 작성

```text
결제 승인 실패 시 중복 재시도를 방지한다.
```

### 2. 타입 선택

잘못된 동작 수정이므로:

```text
fix
```

### 3. scope 선택

결제 영역이므로:

```text
payment
```

### 4. 제목으로 축약

```text
fix(payment): 결제 승인 실패 시 중복 재시도 방지
```

### 5. 본문 필요 여부 판단

다음이 없다면 본문 생략:

- 마이그레이션
- 호환성 변경
- 운영 위험
- 설명이 필요한 기존 동작 차이

---

## 17. 출력 형식

사용자가 커밋 메시지만 요청한 경우에는
설명 없이 바로 사용할 수 있는 메시지를 코드 블록으로 제공한다.

```text
fix(payment): 결제 승인 실패 시 중복 재시도 방지
```

여러 커밋으로 분리해야 하는 경우에는 순서와 메시지를 함께 제시한다.

```text
1. refactor(common): 날짜 범위 검증 로직 분리
2. feat(report): 보고서 조회 기간 검증 추가
3. feat(export): 내보내기 기간 검증 추가
```

실제 커밋까지 요청받은 경우에는:

1. 변경 분석
2. staging 분리
3. 테스트
4. 커밋
5. 결과 보고

순서로 수행한다.

---

## 18. 금지 사항

다음 행동을 하지 않는다.

- 사용자 요청 없이 커밋
- 여러 목적을 하나의 장문 커밋으로 묶기
- 테스트 실패 상태로 커밋
- 제목에 파일명 나열
- 제목에 내부 작업 코드 삽입
- 존재하지 않는 이슈 번호 생성
- `수정`, `반영`, `작업`만으로 제목 작성
- 본문에 테스트 로그 전체 포함
- 본문에 내부 추론 과정 포함
- AI 생성 여부 표시
- `Claude`, `Codex`, `ChatGPT` 등 도구 이름 표시
- 자동으로 `Co-authored-by` 추가
- 사용자의 명시적 요청 없이 push

---

## 19. 최종 체크리스트

커밋 전 아래 항목을 확인한다.

### 커밋 수행 여부

- [ ] 사용자가 실제 커밋을 명시적으로 요청했는가
- [ ] push도 별도로 명시했는가

### Atomic Commit

- [ ] 하나의 목적만 포함하는가
- [ ] 독립적으로 revert 가능한가
- [ ] 독립적으로 cherry-pick 가능한가
- [ ] 해당 커밋 상태에서 빌드 가능한가
- [ ] 구현과 관련 테스트가 함께 있는가
- [ ] 공통 코드는 사용처보다 먼저 반영했는가

### 메시지

- [ ] Conventional Commits 형식인가
- [ ] 타입이 변경 목적과 일치하는가
- [ ] 제목이 한국어인가
- [ ] 제목이 짧고 구체적인가
- [ ] 제목에 마침표가 없는가
- [ ] 제목에 파일 목록이 없는가
- [ ] 제목에 내부 식별자가 없는가
- [ ] 본문이 꼭 필요한 경우에만 작성되었는가
- [ ] 본문이 5~10줄 안팎으로 압축되었는가
- [ ] 파일 목록, 테스트 로그, 추론 과정이 없는가
- [ ] AI 생성 문구가 없는가

## 20. 예시

### 기능 추가

```text
feat(notification): 배송 지연 알림 발송 추가
```

### 버그 수정

```text
fix(cart): 품절 상품 수량 변경 시 오류 방지
```

### 리팩터링

```text
refactor(user): 권한 검증 로직 분리
```

### 문서 변경

```text
docs(readme): 로컬 실행 방법 보완
```

### 본문이 필요한 변경

```text
fix(batch): 실패한 작업만 재처리하도록 변경

전체 배치를 다시 실행하면 성공한 작업까지
중복 처리될 수 있어 실패 상태만 조회한다.

기존 성공 이력과 데이터 형식은 변경하지 않는다.
```

### Breaking Change

```text
feat(api)!: 주문 상태 응답 필드명 변경

클라이언트 응답 구조를 단순화하기 위해
상태 필드명을 통일한다.

BREAKING CHANGE: `orderStatus`를 `status`로 변경한다.
```
