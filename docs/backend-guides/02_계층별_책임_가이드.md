# Spring Backend 계층별 책임 가이드

# 4. 계층별 책임

## 4.1 Command Application Service

Application Service는 유스케이스의 실행 순서를 조정한다.

담당:

- Aggregate 조회
- 도메인 메서드 호출
- 저장소와 외부 Port 조합
- 트랜잭션 경계
- 결과 반환
- 이벤트 발행 시점 조정

담당하지 않음:

- 상태 전이 규칙 직접 구현
- 금액 계산 규칙 직접 구현
- HTTP 파싱
- JPA 쿼리 작성
- Redis 키 문자열 조립
- 외부 API JSON 파싱

```java
@Service
@RequiredArgsConstructor
@Transactional
public class ResourceCommandService {

    private final ResourceRepository resourceRepository;

    public Long create(CreateResourceCommand command) {
        Resource resource = Resource.create(
                command.ownerId(),
                command.name(),
                command.initialValue()
        );

        return resourceRepository.save(resource).getId();
    }

    public void activate(Long resourceId) {
        Resource resource = resourceRepository.findById(resourceId)
                .orElseThrow(() -> new CustomException(
                        ErrorCode.RESOURCE_NOT_FOUND
                ));

        resource.activate();
    }
}
```

나쁜 예:

```java
if (resource.getStatus() == READY) {
    resource.setStatus(ACTIVE);
}
```

좋은 예:

```java
resource.activate();
```

상태 변경 규칙은 도메인 객체가 가진다.

## 4.2 Facade

Facade는 여러 도메인 서비스나 외부 연동을
하나의 사용자 유스케이스로 조합할 때 사용한다.

```java
@Component
@RequiredArgsConstructor
public class ResourceProcessFacade {

    private final ResourceCommandService resourceService;
    private final PaymentPort paymentPort;
    private final AssignmentService assignmentService;
    private final NotificationPort notificationPort;

    public ProcessResult process(ProcessCommand command) {
        Long resourceId = resourceService.create(command.resource());

        PaymentResult payment =
                paymentPort.approve(command.payment());

        AssignmentResult assignment =
                assignmentService.assign(resourceId);

        notificationPort.notifyAssigned(
                resourceId,
                assignment.assigneeId()
        );

        return new ProcessResult(
                resourceId,
                payment.paymentId(),
                assignment.assigneeId()
        );
    }
}
```

사용 기준:

- 둘 이상의 Application Service 호출
- 서로 다른 도메인 조합
- 외부 API 또는 메시지 처리 포함
- 순서가 비즈니스적으로 중요
- 보상 처리 필요

사용하지 않는 경우:

- 단순 CRUD
- 하나의 Aggregate만 변경
- Service를 이름만 바꿔 다시 감싸는 경우

## 4.3 Aggregate

Aggregate Root는 하나의 트랜잭션에서
일관성을 보장할 범위를 대표한다.

```java
@Entity
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
public class Resource {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    private Long ownerId;

    @Enumerated(EnumType.STRING)
    private ResourceStatus status;

    @Embedded
    private Money totalAmount;

    private Resource(
            Long ownerId,
            ResourceStatus status,
            Money totalAmount
    ) {
        this.ownerId = requireOwner(ownerId);
        this.status = status;
        this.totalAmount = totalAmount;
    }

    public static Resource create(
            Long ownerId,
            Money initialAmount
    ) {
        return new Resource(
                ownerId,
                ResourceStatus.CREATED,
                initialAmount
        );
    }

    public void activate() {
        if (!status.canActivate()) {
            throw new CustomException(
                    ErrorCode.INVALID_RESOURCE_STATUS
            );
        }

        status = ResourceStatus.ACTIVE;
    }

    public void cancel() {
        if (!status.canCancel()) {
            throw new CustomException(
                    ErrorCode.RESOURCE_CANNOT_BE_CANCELLED
            );
        }

        status = ResourceStatus.CANCELLED;
    }
}
```

규칙:

- public setter 금지
- 기본 생성자는 `protected`
- 정적 팩터리 또는 의미 있는 생성 메서드 사용
- 외부에서 컬렉션 직접 수정 금지
- 상태 변경은 행동 이름으로 표현
- 생성과 변경 시점 모두 불변식 검증
- 다른 Aggregate는 ID 참조 우선 검토

## 4.4 Value Object

적합한 대상:

- 금액
- 주소
- 좌표
- 기간
- 수량
- 이메일
- 이름
- 소속 조직
- 전화번호
- 거리
- 요청 메모

```java
@Embeddable
@Getter
@NoArgsConstructor(access = AccessLevel.PROTECTED)
@EqualsAndHashCode
public class Money {

    private long value;

    private Money(long value) {
        if (value < 0) {
            throw new IllegalArgumentException(
                    "금액은 음수일 수 없습니다."
            );
        }

        this.value = value;
    }

    public static Money of(long value) {
        return new Money(value);
    }

    public Money add(Money other) {
        return new Money(value + other.value);
    }

    public Money multiply(int quantity) {
        if (quantity <= 0) {
            throw new IllegalArgumentException(
                    "수량은 양수여야 합니다."
            );
        }

        return new Money(value * quantity);
    }
}
```

`@Embeddable` VO의 컬럼명은 VO 내부보다 embedding 하는 Entity에서
`@AttributeOverride`로 지정한다.

```java
@Embedded
@AttributeOverride(
        name = "value",
        column = @Column(name = "total_amount", nullable = false)
)
private Money totalAmount;
```

규칙:

- 가능한 한 불변
- 생성 시 검증
- 값 기준 동등성
- 계산 로직을 VO 내부에 배치
- DTO와 VO를 혼동하지 않음
- JPA에 embedding 되는 VO는 `record`가 아니라 class 기반 `@Embeddable`로 작성
- 기본 형태는 `@Getter`, `@NoArgsConstructor(access = AccessLevel.PROTECTED)`, `@EqualsAndHashCode`, private 생성자, 정적 팩터리 사용
- VO 내부에는 컬럼명을 고정하는 `@Column(name = ...)`을 두지 않음
- 같은 VO를 여러 컬럼명으로 재사용할 수 있도록 컬럼명은 Entity의 `@AttributeOverride`에서 지정
- Entity에는 형식 검증이나 의미가 있는 문자열을 그대로 두지 않고 VO로 감싼다.
- 예: `String email`, `String name`, `String organization` 대신 `AdminEmail`, `AdminName`, `AdminOrganization`

## 4.5 Repository Port

Repository는 다음 세 층으로 분리한다.

- 도메인 저장소 계약: `ResourceRepository`
- 저장소 구현체: `ResourceRepositoryImpl`
- Spring Data JPA 인터페이스: `ResourceJpaRepository`

Application Service는 도메인 저장소 계약에만 의존한다.
`JpaRepository`는 infrastructure/persistence 내부에서만 사용한다.

```java
public interface ResourceRepository {

    Resource save(Resource resource);

    Optional<Resource> findById(Long resourceId);

    Optional<Resource> findByIdForUpdate(Long resourceId);
}
```

도메인 Repository에 화면 전용 조회를 계속 추가하지 않는다.

## 4.6 Persistence Adapter

```text
infrastructure/persistence/
├── ResourceJpaRepository.java
├── ResourceRepositoryImpl.java
├── ResourceJpaEntity.java
└── ResourceMapper.java
```

```java
public interface ResourceJpaRepository
        extends JpaRepository<Resource, Long> {

    @Lock(LockModeType.PESSIMISTIC_WRITE)
    @Query("""
        select r
        from Resource r
        where r.id = :resourceId
    """)
    Optional<Resource> findByIdForUpdate(
            @Param("resourceId") Long resourceId
    );
}
```

```java
@Repository
@RequiredArgsConstructor
public class ResourceRepositoryImpl
        implements ResourceRepository {

    private final ResourceJpaRepository jpaRepository;

    @Override
    public Resource save(Resource resource) {
        return jpaRepository.save(resource);
    }

    @Override
    public Optional<Resource> findById(Long id) {
        return jpaRepository.findById(id);
    }

    @Override
    public Optional<Resource> findByIdForUpdate(Long id) {
        return jpaRepository.findByIdForUpdate(id);
    }
}
```

## 4.7 Query Service와 Repository

조회는 Aggregate 대신 조회 DTO를 직접 반환할 수 있다.

```java
@Service
@RequiredArgsConstructor
@Transactional(readOnly = true)
public class ResourceQueryService {

    private final ResourceQueryRepository queryRepository;

    public ResourceDetail getDetail(Long resourceId) {
        return queryRepository.findDetail(resourceId)
                .orElseThrow(() -> new CustomException(
                        ErrorCode.RESOURCE_NOT_FOUND
                ));
    }

    public Page<ResourceSummary> search(
            ResourceSearchCondition condition,
            Pageable pageable
    ) {
        return queryRepository.search(condition, pageable);
    }
}
```

```java
public interface ResourceQueryRepository {

    Optional<ResourceDetail> findDetail(Long resourceId);

    Page<ResourceSummary> search(
            ResourceSearchCondition condition,
            Pageable pageable
    );
}
```

원칙:

- `@Transactional(readOnly = true)` 기본
- 화면별 조회 DTO 허용
- N+1 문제 제어
- 검색 조건 객체 사용
- Querydsl 구현은 infrastructure에 격리
- 조회 때문에 Aggregate 전체를 억지로 로딩하지 않음

## 4.8 API

Controller는 도메인 내부가 아니라 최상위 `api/` 패키지 아래에 둔다.
도메인별 API는 `api/<domain>/` 아래에 모은다.

예:

```text
api/
└── resource/
    ├── ResourceCommandController.java
    ├── ResourceQueryController.java
    └── dto/
        ├── CreateResourceRequest.java
        └── ResourceDetailResponse.java
```

Controller는 다음만 담당한다.

- HTTP 요청 수신
- 인증 사용자 추출
- 입력 검증
- Request를 Command로 변환
- Application 호출
- 응답 변환

```java
@RestController
@RequiredArgsConstructor
@RequestMapping("/api/resources")
public class ResourceCommandController {

    private final ResourceCommandService resourceService;

    @PostMapping
    public ApiResponse<CreateResourceResponse> create(
            @Valid @RequestBody CreateResourceRequest request,
            @AuthenticationPrincipal LoginUser loginUser
    ) {
        Long id = resourceService.create(
                request.toCommand(loginUser.id())
        );

        return ApiResponse.success(
                new CreateResourceResponse(id)
        );
    }
}
```

Controller 금지 사항:

- Entity 직접 생성
- Repository 직접 호출
- 도메인 상태 직접 변경
- 복잡한 분기
- 외부 API 직접 호출
- 트랜잭션 처리
- 개별 try-catch
