# Spring Backend 서버 간 통신 보안 가이드

## 1. 목적

이 문서는 관리자 서버와 사용자 서버처럼 신뢰된 서버끼리 HTTP API로 통신할 때
적용할 인증, 서명, secret 관리, 로깅 기준을 정리한다.

서버 간 API는 일반 사용자나 관리자 브라우저에서 직접 호출하는 API가 아니다.
따라서 사람 사용자의 JWT 인증과 시스템 간 인증을 분리한다.

---

## 2. 공개 가능한 보안 규격 범위

보안 규격은 원칙적으로 문서화할 수 있다.
인증 알고리즘, 헤더 이름, 서명 대상, timestamp 허용 범위, nonce 재사용 차단,
secret 관리 원칙은 팀원이 구현과 리뷰를 일관되게 하기 위해 문서에 남긴다.

단, 다음 값은 Git에 커밋되는 문서에 쓰지 않는다.

- 실제 API key
- 실제 HMAC secret
- 운영 client id 전체 목록
- 운영 서버 private IP 또는 allowlist 상세
- 운영 도메인별 내부 라우팅 정보
- Redis에 저장되는 실제 nonce key prefix 전체
- 운영 장애 대응용 우회 토큰
- secret rotation 절차에 쓰이는 실제 임시 값

문서에는 dummy 예시만 사용한다.

좋은 예:

```text
X-Internal-Client-Id: demo-user-server
X-Internal-Signature: base64(hmacSha256(secret, canonicalRequest))
```

나쁜 예:

```text
X-Internal-Client-Id: <운영 client id 원문>
secret: <Git 문서에 쓰면 안 되는 운영 secret>
```

판단 기준:

- 공격자가 알아도 secret 없이는 요청을 위조할 수 없는 규칙은 문서화 가능
- secret, 운영망 정보, 우회 수단처럼 알면 바로 공격 난도가 낮아지는 값은 문서화 금지
- 공개 저장소 전환 가능성을 기준으로 작성

## 3. 기본 인증 방식

초기 서버 간 인증은 API Key 식별자와 HMAC 서명을 사용한다.

이 방식은 다음 목적을 가진다.

- 호출 서버 식별
- 요청 위변조 방지
- 재전송 공격 방지
- 관리자 JWT와 시스템 인증 경계 분리

권장 헤더:

```http
X-Internal-Client-Id: demo-user-server
X-Internal-Timestamp: 2026-07-23T22:00:00+09:00
X-Internal-Nonce: 0f70b939-5e1e-4afd-a7e6-c31a3f4c0a46
X-Internal-Signature: base64(hmacSha256(secret, canonicalRequest))
```

규칙:

- `clientId`는 호출 서버를 식별하는 값이다.
- `secret`은 서버별로 다르게 발급한다.
- `signature`는 요청마다 새로 생성한다.
- 동일 secret을 여러 서버가 공유하지 않는다.
- 일반 관리자 JWT를 서버 간 API 인증에 사용하지 않는다.

## 4. Canonical Request

서명 대상 문자열은 서버와 클라이언트가 동일하게 생성해야 한다.

기본 형식:

```text
HTTP_METHOD + "\n" +
REQUEST_PATH + "\n" +
SORTED_QUERY_STRING + "\n" +
X-Internal-Timestamp + "\n" +
X-Internal-Nonce
```

예:

```text
GET
/internal/api/festivals
page=0&size=20&status=UPCOMING
2026-07-23T22:00:00+09:00
0f70b939-5e1e-4afd-a7e6-c31a3f4c0a46
```

정규화 규칙:

- HTTP method는 대문자로 변환한다.
- path는 scheme, host, port를 제외한 request path만 사용한다.
- query parameter는 key 오름차순으로 정렬한다.
- query parameter 값은 URL encoding 결과를 기준으로 맞춘다.
- query parameter가 없으면 빈 문자열을 둔다.
- body가 있는 요청은 추후 `bodyHash`를 canonical request에 추가한다.

GET 조회 API 기준으로는 body를 서명 대상에 넣지 않는다.
POST, PUT, PATCH 내부 API가 생기면 request body SHA-256 hash를 포함하도록 확장한다.

## 5. 검증 절차

서버는 `/internal/api/**` 요청에서 다음 순서로 검증한다.

1. 필수 헤더 존재 여부 확인
2. `clientId` 등록 여부 확인
3. timestamp 파싱 가능 여부 확인
4. timestamp가 현재 시각 기준 허용 범위 안인지 확인
5. nonce가 이전에 사용되지 않았는지 확인
6. canonical request 생성
7. HMAC signature 비교
8. nonce를 Redis에 TTL과 함께 저장
9. 내부 API 실행

권장 timestamp 허용 범위:

```text
현재 시각 기준 5분 이내
```

권장 nonce TTL:

```text
timestamp 허용 범위보다 약간 길게 설정
예: 10분
```

signature 비교는 문자열 단순 비교가 아니라 constant-time 비교를 사용한다.

## 6. Secret 관리

local 개발:

```yaml
app:
  internal-auth:
    clients:
      demo-user-server:
        secret: ${APP_INTERNAL_DEMO_USER_SECRET}
```

규칙:

- 실제 secret은 `application-secret.yml` 또는 환경 변수로만 주입한다.
- `application-secret.yml`은 Git에 커밋하지 않는다.
- 문서와 example yml에는 dummy 값만 둔다.
- 운영 배포에서는 CI/CD secret, 서버 환경 변수, Secret Manager, Kubernetes Secret 중 하나를 사용한다.
- secret은 서버별로 분리한다.
- secret rotation을 고려해 같은 client에 active/previous secret을 둘 수 있다.

운영 secret rotation 권장:

1. 새 secret을 previous가 아닌 active 후보로 배포
2. 호출 서버가 새 secret으로 서명하도록 변경
3. 일정 기간 active/previous 모두 허용
4. 이전 secret 제거

## 7. Replay 공격 방지

timestamp만 검증하면 허용 시간 안에서 같은 요청이 재사용될 수 있다.
따라서 nonce 재사용 차단을 함께 적용한다.

Redis 저장 예:

```text
internal-auth:nonce:{clientId}:{nonce}
```

주의:

- 위 key는 문서용 예시다.
- 운영 key prefix는 설정값으로 관리할 수 있다.
- nonce 저장 실패 시 인증 실패로 처리하는 것을 기본으로 한다.
- Redis 장애 시 내부 API를 열어둘지 차단할지 서비스 중요도에 따라 결정한다.

기본 권장:

```text
Redis 장애 시 내부 API 인증 실패 처리
```

조회 API 가용성이 더 중요하면 사용자 서버 캐시 fallback을 먼저 검토한다.

## 8. 네트워크 보안

HMAC은 요청 위변조를 막기 위한 애플리케이션 계층 보안이다.
운영 환경에서는 네트워크 계층 제한도 함께 적용한다.

권장:

- HTTPS 필수
- private network 또는 같은 VPC 통신 우선
- API Gateway, Load Balancer, Security Group에서 호출 서버 제한
- 운영 환경 IP allowlist 적용 검토
- 외부 공개망에 노출되는 경우 rate limit 적용
- 필요 시 mTLS 적용

주의:

- IP allowlist만으로 인증을 대체하지 않는다.
- HMAC만 믿고 HTTP 평문 통신을 허용하지 않는다.
- 내부망이라는 이유로 secret 검증을 생략하지 않는다.

## 9. API 경로와 Security Filter

내부 API는 일반 API와 경로를 분리한다.

권장:

```text
/internal/api/**
```

Security 구성:

- 내부 API 인증 필터는 `/internal/api/**`에만 적용한다.
- 관리자 JWT 필터와 책임을 분리한다.
- 내부 API는 관리자 로그인 사용자 context를 요구하지 않는다.
- 인증 실패와 인가 실패 ErrorCode를 분리한다.

권장 패키지:

```text
global/security/internal/
├── InternalApiAuthenticationFilter.java
├── InternalApiSignatureVerifier.java
├── InternalApiClientProperties.java
└── InternalApiNonceRepository.java
```

## 10. 로깅 기준

로그에 남겨도 되는 값:

- request path
- method
- client id 일부 또는 해시
- 인증 실패 사유의 범주
- timestamp 파싱 성공 여부
- nonce 재사용 여부

로그에 남기면 안 되는 값:

- secret
- signature 전체 값
- authorization header
- 운영 internal endpoint 전체 구성 정보
- 개인정보

signature가 필요하면 앞 6자리 정도만 마스킹해서 남긴다.

```text
signature=abc123****
```

## 11. 테스트 기준

필수 테스트:

- 정상 서명 인증 성공
- 필수 헤더 누락 실패
- 미등록 client id 실패
- timestamp 만료 실패
- timestamp 미래 시간 실패
- nonce 재사용 실패
- signature 불일치 실패
- query parameter 정렬 차이 검증
- URL encoding 포함 query 검증
- secret 미설정 시 애플리케이션 시작 또는 인증 실패 정책 검증
- 로그에 secret/signature 전체 값이 남지 않는지 검토

경계값:

- timestamp 허용 범위 정확히 안쪽
- timestamp 허용 범위 정확히 바깥쪽
- 빈 query string
- 최대 허용 nonce 길이
- 최대 허용 client id 길이

## 12. 적용 판단 기준

API Key + HMAC이 적합한 경우:

- 호출 서버 수가 적다.
- 서버 간 요청이 단순하다.
- 별도 인증 서버를 두기에는 이르다.
- 관리자 JWT와 시스템 인증을 명확히 분리하고 싶다.

OAuth2 client credentials 또는 internal JWT를 검토할 경우:

- 호출 서버가 많아진다.
- 권한 scope가 복잡해진다.
- 중앙 인증 서버가 필요하다.
- secret rotation과 권한 폐기를 중앙에서 관리해야 한다.

mTLS를 검토할 경우:

- 운영망 보안 요구가 높다.
- 인증서를 배포하고 갱신할 운영 체계가 있다.
- 네트워크 레벨에서 서버 신원을 강하게 검증해야 한다.

## 13. 사용자 서버 축제 목록 조회 API 적용 메모

사용자 서버가 관리자 서버에서 진행 예정, 진행 중, 진행 완료 축제 목록을 조회하는 API는
서버 간 API로 분류한다.

적용 기준:

- 경로는 `/internal/api/festivals` 계열 사용
- 인증은 이 문서의 API Key + HMAC 방식 적용
- 응답은 사용자 서버에 필요한 DTO projection만 반환
- 관리자 JWT 인증과 분리
- 사용자 앱에서 직접 호출하지 않음
- 사용자 서버에서는 짧은 TTL 캐시를 검토

현재 `FestivalStatus`가 `DRAFT`만 있으므로 사용자 서버 노출 상태는 우선 날짜 기준
`progressStatus`로 계산한다.
